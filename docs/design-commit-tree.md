# Design Addendum: Compact Commit Tree Context

Generated by office-hours on 2026-03-26
Project: patchdeck
Status: DRAFT
Mode: Builder
Extends: [design.md](./design.md)

## Problem Statement

`patchdeck`의 현재 `COMMITS` pane은 브랜치 commit 목록과 commit/file diff drill-down까지는 제공한다. 하지만 commit들이 어떤 줄기에서 이어지는지, 어디서 merge가 들어오는지, 현재 선택한 commit이 브랜치 흐름 안에서 어떤 위치인지 한눈에 파악하기는 어렵다.

목표는 `patchdeck`를 full Git client로 바꾸지 않으면서도, commit 목록 안에 **compact tree context**를 기본값으로 제공해 review 맥락을 더 빠르게 읽게 하는 것이다.

## What Makes This Cool

- commit subject를 읽기 전에 "이건 merge 줄기인가, 직선 흐름인가"를 바로 안다.
- `git log --graph`의 정보 가치를 가져오되, `patchdeck`다운 calm review console 미학은 유지한다.
- commit tree는 별도 장식이 아니라 diff review 우선순위를 잡는 보조 시각 언어가 된다.

## Constraints

- `patchdeck`의 본질은 review console이지 full Git client가 아니다.
- graph는 commit subject보다 앞서면 안 된다. subject가 계속 주인공이어야 한다.
- v1 graph 범위는 현재 worktree branch의 최근 commit들과 merge 문맥까지만 다룬다.
- 기본값으로 항상 보여야 하지만, 화면을 noisy하게 만들면 안 된다.
- commit 전체 diff, commit file 목록, file diff drill-down과 충돌하면 안 된다.

## Premises

1. 이 기능의 목적은 "브랜치/merge 맥락을 빨리 읽는 것"이지, Git topology 전체를 탐색하는 것이 아니다.
2. graph는 `COMMITS` pane 안의 **inline tree strip** 형태여야 하며, 별도 큰 pane이나 full canvas는 과하다.
3. 구현은 단순 파싱 어댑터보다 `Commit Graph Model Layer`로 분리해야 장기적으로 `patchdeck` 성격을 지킬 수 있다.
4. v1에서 remote branches, tags, reflog, stash까지 끌고 오면 review console이 아니라 Git browser가 된다.

## Approaches Considered

### Approach A: Inline Tree Strip

commit 목록 각 줄 앞에 간단한 graph glyph를 붙이고, 현재 commit 로딩 로직 위에 최소한의 graph metadata만 추가한다.

장점:
- 가장 빨리 붙는다.
- 현재 UI 구조를 거의 유지한다.

단점:
- 내부 로직이 점점 TUI 파일로 새어 나갈 수 있다.
- 이후 확장 시 구조가 쉽게 지저분해진다.

### Approach B: Commit Graph Model Layer

commitmodel 안에 parent/merge 관계, line segment, glyph hint를 계산하는 graph layer를 만들고, TUI는 그 결과를 렌더링만 한다.

장점:
- 구조가 가장 깔끔하다.
- tree context를 다른 surfaces(list mode, future export, tests)에도 재사용할 수 있다.
- `patchdeck`를 Git client로 키우지 않고도 내부 책임 분리가 유지된다.

단점:
- v1 기준 구현량이 더 크다.
- graph를 어디까지 단순화할지 초기에 명확히 못 박아야 한다.

### Approach C: Git Output Adapter

`git log --graph --oneline` 출력을 직접 받아 partial parse해서 현재 commit list에 맞게 보여준다.

장점:
- 빠르게 prototype이 가능하다.
- Git의 실제 graph 감각을 즉시 얻는다.

단점:
- 출력 파싱이 brittle하다.
- selection, commit-file drill-down, diff 상태와 결합될수록 제어권이 약하다.

## Recommended Approach

선택한 방향은 **Approach B: Commit Graph Model Layer** 다.

핵심 이유:

- `patchdeck`는 이미 worktree, current file diff, branch commit diff, commit file diff까지 흐름이 생겼다.
- 이 시점에서 graph까지 TUI 안에서 임기응변으로 붙이면, 지금은 빨라 보여도 이후 유지보수성이 급격히 나빠진다.
- 내부적으로는 graph model을 분리하고, 화면에는 그 결과를 아주 얇게만 노출하는 것이 `patchdeck`답다.

## Graph Model Scope

### v1에 포함

- 현재 worktree branch의 최근 commit들
- merge commit 문맥
- inline glyph vocabulary: `*`, `|`, `\`, `/`
- commit row 앞의 compact tree strip
- merge commit과 직선 commit을 빠르게 구분할 수 있는 최소 line context
- 최대 50개 commit 범위 안에서만 graph를 계산한다
- parent가 50개 범위 바깥에 있으면 graph는 그 지점에서 끊긴 것으로 표시한다

### v1에서 제외

- local branch 전체 topology 시각화
- remote branch 전체 graph
- tag, stash, reflog, detached HEAD history view
- zoomable graph canvas
- graph 자체를 탐색하는 별도 mode
- octopus merge의 완전한 topology 재현
- rebase/cherry-pick 관계를 별도 시각 언어로 구분하는 기능

## Proposed Internal Model

```text
Commit
  - hash
  - shortHash
  - subject
  - relativeTime
  - parents []string

CommitGraphRow
  - commit Commit
  - columns []GraphCell
  - glyph string
  - isMerge bool
  - continuation: none | out_of_range

GraphCell
  - kind: empty | vertical | merge_in | merge_out | node
```

설계 원칙:

- commitmodel은 Git 명령 출력에서 commit과 parent 관계를 읽는다.
- v1 Git contract는 아래를 따른다:

```text
git log --date-order --format=%H%x1f%h%x1f%ct%x1f%P%x1f%s -n 50 <range>
```

- parser는 `hash`, `shortHash`, `unixTime`, `parents(raw %P split)`, `subject`를 읽는다.
- parent가 현재 50개 범위 안에 없으면 `out_of_range_parent`로 간주하고, graph row는 더 이상 이어 그리지 않는다.
- graph layer는 commit list를 받아 row별 glyph metadata를 계산한다.
- TUI는 graph layer가 준 결과를 그릴 뿐, branch topology를 직접 계산하지 않는다.

## Graph Rendering Rules

- subject는 항상 commit row의 주인공이다.
- graph glyph는 subject보다 앞에 오지만, **preattentive hint**일 뿐 시각적 우선순위는 subject보다 낮다.
- merge commit은 "먼저 glyph로 감지되고, 확인은 subject로 한다"가 정확한 목표다.
- graph column 수는 최대 4개로 제한한다. 이 한도를 넘는 복잡한 topology는 단순화한다.
- narrow terminal에서는 graph strip를 1~2 glyph 수준으로 축약한다.
- graph strip가 subject 가독성을 해치면 subject를 우선하고 graph를 줄인다.
- `out_of_range_parent`는 별도 가짜 row를 추가하지 않는다. 현재 commit row 안에서 마지막 graph column을 `|` 또는 `…`에 해당하는 continuation glyph로 축약해 표시한다.

ASCII examples:

Linear history:

```text
* a1b2c3d feat: add commit file drill-down
| d4e5f6a fix: keep diff target in sync
| f7g8h9i feat: add commit list mode
```

Single merge:

```text
*   abcd123 Merge branch 'origin/master'
|\  9abc456 feat: add tree strip
|/  7def890 fix: commit selection sync
```

Truncated / out-of-range parent:

```text
* abc1234 Merge branch 'origin/master'
| abc1234 continues beyond loaded history
```

## UI Behavior

- `COMMITS` pane의 각 행 맨 앞에 compact glyph strip를 둔다.
- glyph strip는 subject보다 시각적으로 조용해야 한다.
- merge commit은 subject 자체보다 앞의 glyph 문맥으로 먼저 읽히게 한다.
- 선택 상태(`>`)와 graph glyph는 서로 싸우면 안 된다.
  - selection이 가장 강한 신호
  - graph는 한 단계 낮은 신호
- `COMMIT FILES`와 `DIFF` drill-down 흐름은 그대로 유지한다.
- graph를 켠다고 해서 pane 수가 늘어나지 않는다. 여전히 `COMMITS` pane 내부 표현의 문제다.

Responsive behavior:

- wide: commit subject 앞에 full compact strip(최대 4 columns)
- medium: 2-3 columns로 축약
- narrow: 1-2 columns만 남기고 merge 감지만 유지
- graph 계산 실패 시 `COMMITS` pane은 기존 리스트 렌더링으로 fallback 한다. graph 때문에 commit browsing이 깨지면 안 된다.
- failure 조건은 아래로 제한한다:
  - parent parse 실패
  - graph column cap(4)을 넘는 unsupported topology
  - terminal width가 graph strip 최소 표현 폭보다 좁은 경우

## Open Questions

- merge commit subject를 따로 badge 처리할지, graph glyph만으로 충분한지
- 매우 좁은 terminal에서 glyph strip를 어느 수준까지 줄일지
- `--list` mode에는 넣지 않는다. v1 graph는 TUI 안에서만 제공한다.

## Success Criteria

- 사용자가 commit subject를 다 읽지 않아도 merge/직선 흐름을 빠르게 구분할 수 있다.
- commit tree strip를 추가해도 `COMMITS` pane의 subject 가독성이 크게 떨어지지 않는다.
- `COMMITS -> COMMIT FILES -> DIFF` drill-down과 graph 표현이 충돌하지 않는다.
- graph 계산 로직이 TUI 파일 밖의 모델 레이어에 남아 있어 테스트 가능하다.

## Next Steps

1. `commitmodel`에 `%P` 기반 parent metadata를 추가한다.
2. `CommitGraphRow`를 계산하는 graph layer를 만든다.
3. `COMMITS` pane renderer를 graph-aware row renderer로 교체한다.
4. wide/medium/narrow breakpoints별 glyph 축약 규칙을 넣는다.
5. parser unit tests, graph row golden tests, manual dogfooding을 돌린다.

## Delivery Phases

### Phase 1: Parser / Model

- `%P` 포함 Git log contract 도입
- commit + parent parsing
- graph row builder
- unit tests / golden tests

### Phase 2: Renderer

- `COMMITS` pane row renderer 교체
- responsive 축약 규칙 적용
- merge-heavy branch manual verification

이 기능은 "thin UI tweak"가 아니라 모델과 렌더러를 모두 건드리는 변경이므로, 두 단계로 나눠 진행한다.

## Verification

- parser unit tests: parent parsing, out-of-range parent handling
- golden tests: linear history, single merge, truncated merge context
- manual checks:
  - wide terminal
  - medium terminal
  - narrow terminal
  - merge-heavy branch
  - graph failure fallback

## What I noticed about how you think

- 당신은 "commit 목록도 보고 싶다"에서 멈추지 않고, `"git log tree 형태로 볼 수 있게"`라고 말하면서 단순 리스트의 한계를 정확히 짚었다.
- full graph pane보다 `patchdeck`다운 흐름을 유지해야 한다는 점에 동의하면서도, `tree context`는 기본값이어야 한다고 밀어붙인 건 좋은 판단이었다.
- 구현 방식도 즉흥적인 adapter보다 `Commit Graph Model Layer`를 고른 걸 보면, 지금 필요한 기능보다 다음 확장까지 같이 보고 있다.
